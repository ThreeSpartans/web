Apollo Platform

Introduction

Apollo is the codename for the library of reusable UI and non-UI components that can be used across various mobile apps developed in Conduent CTI Mobility. It will be helpful in acheiving

 

  1. Single code base
  2. Common UI and behaviour
  3. Acceleration in mobile app development
  4. New enhancements flow across all projects seamlessly
  5. Reduced number of issues
  6. Standard across CTI Mobility

Development Guidelines

  • The name of the components should be
    • Prefixed by CM (which could be an Abbreviation for Conduent (CTI) Mobility)
    • Similar to the name of the similar component available in Platforms SDK
    • If no similar component is available in SDK, then should follow the name of the other platform
    • If there is no similar component is both platforms, then all the platforms should use the same name
  • The attributes/properties/methods/listeners/delegates should be similar to the component available in platform SDK ex: CMButton should be used for Button
  • Try to make as many properties/attributes as possible configurable
  • If any existing hard-coded property is made configurable, then the default value should be the previously hard-coded value to support backward compatibility
  • Even if a native component need to be used without any customization, it should still be created as a CM component without any customization and used 

Development Methodology

  • Test Driven Development
    • If TDD is not possible, then Unit tests can be written later
  • 100% code coverage (if less than 100%, then after approval from Architect)
    • public attributes and properties
    • methods
    • delegates, listeners
    • ensure default values for all the properties/attributes
    • internal methods having complex logic
  • Low level design to be followed - Detailed description of the following should be formally documented
    • Class Names
    • Member variables along with 
      • Access modifiers and
      • Comments on usage
    • Methods with details about
      • Access modifiers
      • Comments on usage
      • Input parameters and their usage
      • Output parameters and their usage
  • Branching Strategy - Developer branches (Since a developer will be working on 1 component at a time most often, feature branches wont make sense)
  • Branching Policy -TBD
  • Code commit policy for PUSHing to master branch
    • Make sure no warnings are there
    • Make sure all backward compatibility is met
    • Memory Profiling
    • Proper comments are there
    • Follow general coding standards and guidelines
    • Self review of
      • Code
      • Unit Test cases
      • Comments
      • LLD
    • Peer Code review
      • Code
      • Unit Test cases
      • Comments
      • LLD
    • Architect Code review
      • Code
      • Unit Test Cases
      • Comments
      • LLD

Guidelines on Structure

Apollo will be structured based on the following hierarchy

  • Apollo Core - This will include all the UI and non-UI controls if all of the following are true
    • the usage probability in at least 75% of projects is at least 75%
    • is small limited to less than approx 500 lines of code
    • after approval from architect
  • Other specific components which cannot fit in Core like the following either due to less probability of usage or big size
    • Bar Code
    • Networking
    • Calendar
    • Keypad

All the components should follow the below guidelines

  • Should be created as Library/Framework/POD/Swift Packages and will sit in parallel to Apollo Core

Demo Project

There will be a common demo project. All the components should be shown as examples in the demo. The demo mainly has a 2 levels.

  • Level 1 - First screen will be list of all the components shown in a list view
  • Level 2 - when any list item is clicked, it will take to the respective component's example page
    • On that page, show various customization of the component one after the other similar to what we have for EditText in Android
